
# Data Security & Audit Framework

## Purpose

This document defines the security and audit architecture used to protect borrower and investor data within the lending platform.

Because the platform processes:

- financial data
- personally identifiable information (PII)
- investment transactions

a strong audit framework is required.

---

# Security Principles

The platform is built on the following principles:

- Least privilege access
- End-to-end encryption
- Immutable financial records
- Full audit trails
- Separation of duties
- Defense-in-depth — multiple independent enforcement layers so a failure in one does not compromise the system

---

# Defense-in-Depth Implementation

Six enforcement layers are applied to every request. Each layer is independent — a bypass at one layer is caught by the layers below.

## Layer 1 — Infrastructure (Vercel + Supabase + Upstash)
- Vercel spend cap prevents runaway billing from volumetric attacks
- Supabase budget alert triggers at threshold overage
- Upstash daily command limit caps Redis usage

## Layer 2 — Middleware (proxy.ts, Next.js Edge)
- IP-based rate limiting on public endpoints (signup, forgot-password) via Upstash Redis
- Authentication presence enforced — unauthenticated requests redirected to `/login`
- Role-based route access — wrong role redirected to own dashboard
- Open redirect protection on `/auth/callback` — only relative paths allowed for `next` param

## Auth Callback Routes

Two server-side routes handle all post-authentication redirects. Raw JWTs are never exposed in the URL or browser history.

| Route | Handles | Method |
|---|---|---|
| `/auth/confirm` | Invite emails, password reset | `verifyOtp(token_hash)` — Supabase email template sends a hashed token as a query param; verified server-side |
| `/auth/callback` | Magic links, OAuth | `exchangeCodeForSession(code)` — PKCE flow; browser stores code verifier in cookies, exchange is server-side |

**Invite flow uses token_hash** because invites are server-initiated — no browser session exists when the admin sends the invite, so PKCE code verifier generation is not possible. The hashed token is useless without Supabase's server-side secret.

**Magic links and OAuth use PKCE** (`flowType: 'pkce'` on the browser client). An intercepted redirect URL is useless without the code verifier stored in the user's cookies.

## Layer 3 — API Route Handlers
Every API route enforces in this exact order:
1. **Zod validation** — malformed or out-of-bounds input returns 400 before Supabase is called
2. **Per-user rate limiting** — Upstash Redis counter by user ID, returns 429 if exceeded
3. **Authentication** — `supabase.auth.getUser()` returns 401 if session invalid
4. **Role authorization** — `getUserRole()` queries `user_roles` table, returns 403 if insufficient

## Layer 4 — Supabase RLS
Row-level security is enabled on all tables. Policies are enforced at the PostgreSQL layer regardless of what the API layer permits. The `is_admin()` helper function queries the `user_roles` table — not JWT metadata.

## Layer 5 — Drizzle ORM
All queries use parameterized statements generated by Drizzle. SQL injection is prevented by construction.

## Layer 6 — Budget Monitoring
Vercel, Supabase, and Upstash are each configured with spend caps or budget alerts to detect abnormal usage volumes.

# Rate Limiting Architecture

Upstash Redis is used as the rate-limit counter store. It is serverless-native and Edge-compatible — counters persist across all serverless function invocations without requiring a persistent connection.

**Public endpoint limits (IP-based):**

| Endpoint | Limit | Window | Algorithm |
|---|---|---|---|
| Signup | 5 requests | 10 min | Sliding window |
| Forgot password | 3 requests | 15 min | Sliding window |

**Authenticated endpoint limits (user ID-based):**

| Endpoint | Limit | Window | Algorithm |
|---|---|---|---|
| Submit application | 5 requests | 1 hour | Fixed window |
| Invite user | 20 requests | 1 hour | Fixed window |
| Status / metrics update | 60 requests | 1 hour | Fixed window |

Sliding window is used for public endpoints to prevent burst resets at window boundaries — critical for credential-stuffing and email-enumeration prevention.

# Role-Based Access Control

Roles are stored in the `user_roles` PostgreSQL table — not in JWT metadata. JWT claims can be forged or replayed; database rows cannot.

**Role seeding:**
- Public signup: `handle_new_user()` trigger forces role to `borrower` regardless of metadata
- Invite flow: admin API route seeds `user_roles` immediately on invite creation; trigger seeds again on acceptance
- Role validation against `CHECK` constraint — no invalid role can be inserted

**Role enforcement:**
- Middleware (`proxy.ts`): queries `user_roles` for route access decisions
- API routes: `getUserRole()` helper queries `user_roles` before every authorization check
- RLS: `is_admin()` and `get_user_role()` functions query `user_roles` at the database layer

# Financial Ledger Integrity

The system should maintain a **double-entry accounting model**.

Each financial transaction contains:

Debit entry  
Credit entry

Example:

Borrower payment:

Debit: Borrower account  
Credit: Investor receivable account

This ensures the ledger always balances.

---

# Immutable Records

Financial transactions must be append-only.

Rules:

- transactions cannot be deleted
- corrections require reversing entries
- audit history remains permanent

---

# Database Integrity

Recommended databases:

- PostgreSQL
- Microsoft SQL Server

- ACID compliance

Loan documents should be stored using secure object storage.

# Payment Security

Payments must be processed using regulated providers.

Examples:

- Stripe
